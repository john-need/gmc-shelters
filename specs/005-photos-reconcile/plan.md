# Implementation Plan: Photos Tab Reconcile

**Branch**: `005-photos-reconcile` | **Date**: 2026-05-26 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/005-photos-reconcile/spec.md`

## Summary

Add a "Reconcile" modal to the Photos tab that compares the `shelters/{slug}/photos/` filesystem directory against the `photos` SQLite table for the currently selected shelter. It surfaces two lists — image files on disk with no DB record (Untracked Files) and DB records whose files are absent from disk (Orphaned Records) — and lets the operator selectively register or delete items. The modal auto-scans on open, applies changes best-effort on confirm, and reloads the photos list on close. All changes are DB-only; no files are written or deleted from disk.

## Technical Context

**Language/Version**: TypeScript 5, Electron 32, Node.js 20 (main process), React 18 (renderer)  
**Primary Dependencies**: better-sqlite3 (SQLite), Redux Toolkit (renderer state), Electron IPC (main↔renderer bridge), Node `fs/promises` (filesystem scan), Vite (bundler)  
**Storage**: SQLite at `database/gmc_shelters.sqlite` via `src/main/db/`; photo files under configurable `SHELTERS_ROOT/{slug}/photos/`  
**Testing**: Jest with two projects — `main` (Node environment, `src/main/**/*.test.ts`) and `renderer` (jsdom, `src/renderer/**/*.test.tsx`); test files are colocated with source  
**Target Platform**: macOS/Windows desktop Electron app; single-user, local filesystem only  
**Performance Goals**: Scan and display discrepancies within 3 seconds for up to 500 photos (SC-001)  
**Constraints**: DB-only mutations; no file writes or deletes during reconcile; best-effort apply (not all-or-nothing); photos list reloads on modal close; rerun-safe (no duplicate inserts)  
**Scale/Scope**: One shelter at a time; typically 10–200 photos per shelter; up to 500 per SC-001

## Phase 1: Design

### IPC Layer Design

Two new IPC operations are added, following the existing pattern in `src/main/ipc/photos.ts`:

**Scan** (`PHOTOS_RECONCILE_SCAN`):
- Input: `{ shelterId: number; sheltersRoot: string }`
- Main process: fetches the shelter slug, reads `shelters/{slug}/photos/` dir, queries `photos` table, computes the two diff lists
- Output: `ReconcileScanResult` — two arrays of items (see `data-model.md`)

**Apply** (`PHOTOS_RECONCILE_APPLY`):
- Input: `ReconcileApplyInput` — arrays of filenames to add and record IDs to delete
- Main process: for each filename → `insertPhoto`; for each ID → check if default → `clearDefaultPhoto` if needed → `deletePhoto`; collects per-item outcomes
- Output: `ReconcileApplyResult` — counts of added/deleted/failed with brief reason per failure

### Filesystem Scan Design

`listPhotosDir(slug, sheltersRoot)` in `src/main/fs/photos.ts`:
- Uses `fs.promises.readdir` on `photosDirForSlug(slug, sheltersRoot)`
- Filters by recognised extensions: `.jpg`, `.jpeg`, `.png`, `.tif`, `.tiff`, `.webp` (case-insensitive)
- Returns bare filenames (not full paths), matching the format inserted by `copyPhotoToShelter`
- If directory does not exist: returns `[]` (graceful empty — no throw)

### Filename Matching Design

When comparing filesystem filenames to DB `file_name` values:
- Normalise each DB `file_name` by stripping the `shelters/` prefix (reusing the same logic as `photoFilePath()`)
- Then strip `{slug}/photos/` prefix to obtain the bare filename
- Compare case-insensitively against the filesystem list
- A DB record is orphaned if its normalised bare filename is not in the filesystem set
- A filesystem file is untracked if its basename is not in the DB normalised filename set

### Default Photo Clearing Design

`clearDefaultPhoto(shelterId, photoId)` in `src/main/db/photos.ts`:
- Checks `SELECT default_photo_id FROM shelters WHERE id = ?`
- If `default_photo_id === photoId`, runs `UPDATE shelters SET default_photo_id = NULL WHERE id = ?`
- Called from the apply handler before `deletePhoto(id)` for each selected orphaned record

### Renderer State Design

- Reconcile UI state (modal open, scan results, selection state, apply results) lives in **local component state** within `PhotosTab.tsx` — it is transient and does not need to survive navigation
- After apply completes, the modal shows the summary; on dismiss, `dispatch(loadPhotos(shelterId))` refreshes the global Redux store, which re-renders the photos list
- A new `addPhotosLocal` reducer action is added to `photosSlice` to support optimistic insertion if needed; `reconcileApply` thunk calls `loadPhotos` on success for a clean reload

### Modal Component Design

`ReconcileModal` lives in `PhotosTab.tsx` (or a separate `ReconcileModal.tsx` if the component exceeds ~150 lines):
- **Scan phase**: auto-triggered on mount; shows a loading spinner; on completion shows the two sections
- **Select phase**: two labelled sections ("Files not in database" / "Records with no file"), each with Select All / Deselect All and per-item checkboxes; Reconcile button disabled when nothing selected
- **Results phase**: after apply, replaces the lists with a summary ("2 added · 1 deleted · 1 failed"); single Close button
- **Clean state**: when both sections are empty after scan, shows a single "All photos are in sync" message with a Close button
