# Implementation Plan: Photo Editor Dialog

**Branch**: `006-photo-editor-dialog` | **Date**: 2026-05-26 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/006-photo-editor-dialog/spec.md`

## Summary

Move all image-editing controls (crop, rotate, flip, zoom) out of the small right-aside detail panel in the Photos tab and into a full-screen modal dialog. The dialog opens when the user single-clicks the right-aside photo preview or double-clicks a photo card in the grid/list view. Users click Save (image edits only) or Cancel (discards all changes) to close. The implementation is entirely renderer-side: a new `PhotoEditorDialog` component extracted from `PhotosTab`, CSS additions for the full-viewport dialog layout, and updated tests. No IPC contracts, database schema, or operator-facing workflows change.

## Phase 1: Design

See [data-model.md](data-model.md) for entity details. Key design points:

### PhotoEditorDialog component

- **Props**: `photo`, `photoUrl`, `shelterId`, `sheltersRoot`, `isDefault`, `onSave`, `onCancel` (see data-model.md for full contract)
- **Internal state**: `rotation`, `flipped`, `cropping`, `cropRect`, `crop`, `zoom`, `saving` — all transient, reset on unmount
- **Layout**: Full-viewport overlay (`.modal-bg`) containing a two-column layout — large photo preview on the left, compact tool column on the right — or a stacked layout on narrow viewports
- **Toolbar**: Rotate left, rotate right, flip, crop mode toggle, zoom controls — identical logic to current inline tools
- **Footer**: Save button (disabled while `saving`; calls `savePhotoMetadata` then `onSave`) and Cancel button (calls `onCancel`)
- **Focus trap**: `useEffect` attaches keydown listener on mount; traps Tab/Shift+Tab; Escape fires `onCancel`; focus returns to `triggerRef` on close
- **Error handling**: On save failure, show toast via `dispatch(showToast(...))`, set `saving = false`, keep dialog open (FR-011)

### PhotosTab modifications

- Remove state: `rotation`, `zoom`, `flipped`, `cropping`, `crop`, `cropRect`, `naturalSize`, `frameSize` (moved into `PhotoEditorDialog`)
- Remove handlers: `startCropDrag`, `handleSaveEdits` (moved into `PhotoEditorDialog`)
- Keep state: `selectedId`, `view`, `repoRoot`, `detailWidth`, `resizing`, `reconcileOpen`, `dragOver`, `version`
- Add state: `editorOpen: boolean` (controls whether `PhotoEditorDialog` is mounted)
- Photo preview in right-aside: add `onClick={() => setEditorOpen(true)}` + hover CSS (pointer cursor + dark overlay) via new `.photo-preview-clickable` class
- `PhotoCard` / `ListRow`: add `onDoubleClick` prop → `setSelectedId(p.id); setEditorOpen(true)`
- On `onSave`: `setEditorOpen(false); setVersion(v => v + 1)` (bumps cache-busting version for the photo thumbnail)
- On `onCancel`: `setEditorOpen(false)`

### Test plan (written first — constitution compliance)

**`PhotoEditorDialog.test.tsx`** (new, written before implementation):
- Renders dialog with photo preview and all editing controls visible
- Rotate left/right buttons update rotation state (visual via aria-label or data-testid)
- Flip button toggles flip state
- Crop button enters/exits crop mode
- Save button dispatches `savePhotoMetadata` with correct args and calls `onSave` on success
- Save button disabled while saving; shows loading state
- Save failure: dispatch rejects → toast shown, dialog remains open, Save re-enabled
- Cancel button calls `onCancel` without dispatching save
- Escape key calls `onCancel`
- Click on overlay calls `onCancel`
- No pending edits + Save: calls `onSave` without dispatching (or dispatches with zero deltas)

**`PhotosTab.test.tsx`** (updated):
- Right-aside photo preview click opens editor dialog (dialog element appears in DOM)
- Double-click on a grid photo card opens editor dialog
- Double-click on a list row opens editor dialog
- Inline crop/rotate/flip controls are NOT present in the tab without the dialog open
- Editing tools appear inside the dialog, not the aside panel
